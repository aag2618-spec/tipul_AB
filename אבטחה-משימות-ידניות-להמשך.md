### 1. **C2 — Prisma Migration (קריטי, דורש backup)**

**הבעיה:** `package.json` `start:prod` ו-`render.yaml` `startCommand` עדיין משתמשים ב-`prisma db push --accept-data-loss`, שיכול למחוק עמודות/טבלאות בלי אזהרה.

**למה לא בוצע אוטומטית:** דורש backup DB מלא + פעולה ידנית ב-Render Shell. כל טעות יכולה להוביל לאובדן נתונים.

**צעדים:**
1. **בצע backup ב-Render:**
   - Render Dashboard → Database (PostgreSQL) → Backups → Download latest
2. **צור baseline migration לוקאלית:**
   ```powershell
   npx prisma migrate dev --name baseline --create-only
   ```
3. **התחבר ל-Render Shell** (Web Service → Shell):
   ```bash
   npx prisma migrate resolve --applied baseline
   ```
4. **שנה את `package.json`:**
   - מ: `"start:prod": "prisma db push --accept-data-loss && ..."`
   - ל: `"start:prod": "prisma migrate deploy && ..."`
5. **שנה את `render.yaml`:**
   - מ: `startCommand: npx prisma db push --accept-data-loss && ...`
   - ל: `startCommand: npx prisma migrate deploy && ...`
6. **הוסף ל-build:** `prisma migrate status` כדי לזהות drift.

**זהירות:** בקבצי `prisma/sql/*.sql` (4 קבצים) יש raw SQL שלא בוצע כ-migration. צריך לאסוף אותם ל-baseline migration.

### 2. **H2 — JWT Cache Invalidation (גבוהה, מורכב)**

**הבעיה:** `src/lib/auth.ts:13` משתמש ב-cache של 30s ל-JWT decoding. אם משתמש נחסם או role שלו השתנה, ההחלטה תקפה רק עד 30s אחרי השינוי.

**למה לא בוצע אוטומטית:** דורש שינוי schema (טבלת `auth_cache_invalidations` או `User.cacheInvalidatedAt`) + מיגרציה.

**הפתרון המוצע (כשתחליט להטמיע):**
1. הוסף ל-`User` model: `cacheInvalidatedAt DateTime?`
2. בכל `block`/`unblock`/`role change` — עדכן את הזמן.
3. ב-`requireAuth` — אם `token.iat < user.cacheInvalidatedAt` → דחה ודרוש login מחדש.
4. כך ניתן לבטל cache בלי להמתין 30s.

### 3. **H9 — CRM API Key Public Exposure (דורש בדיקה)**

**הבעיה:** `NEXT_PUBLIC_CRM_API_KEY` חשוף ב-bundle הלקוח (layout.tsx:53 + lib/client-hub.ts:16).

**למה לא בוצע אוטומטית:** **דורש בדיקה אם זה widget public key (קונציונאלית public, כמו Intercom) או secret credential.**

**צעדים:**
1. **בדוק בdocumentation של ה-CRM:**
   - האם ה-API key הזה מיועד להיות "publishable" (קריאה בלבד) או "secret" (גישה מלאה)?
   - לדוגמה: Stripe יש `pk_*` (public) ו-`sk_*` (secret).
2. **אם public:** השאר כפי שהוא + הוסף הערה ב-comment.
3. **אם secret:**
   - הסר `NEXT_PUBLIC_` מ-env vars
   - צור `/api/crm/[...slug]/route.ts` proxy server-side
   - עדכן `src/lib/client-hub.ts` לקרוא דרך ה-proxy
   - שנה את `layout.tsx`: השתמש ב-server-rendered token זמני במקום data-api-key
4. **בדוק שווידג'ט הצ'אט עדיין עובד אחרי השינוי.**

## 📋 משתני סביבה ש-validation החדש דורש

לאחר ה-push של `8e7999d`, אם אחד מהמשתנים האלה חסר/לא תקין ב-Render → ה-startup ייכשל:

| Variable | דרישה | איך לייצר |
|---|---|---|
| `NEXTAUTH_SECRET` | ≥32 chars | `openssl rand -base64 32` |
| `ENCRYPTION_KEY` | ≥32 chars (production) | `openssl rand -hex 32` |
| `CRON_SECRET` | קיים (production) | `openssl rand -base64 32` |
| `SETUP_SECRET` | ≥32 chars (אם `SETUP_ENABLED=true`) | `openssl rand -base64 32` |

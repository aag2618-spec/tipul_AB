## 🔧 תיקונים שבוצעו (8 שינויי קוד)

### ✅ תיקון 1 — Security Headers
**קובץ:** [next.config.ts](next.config.ts)
- הוספת `Strict-Transport-Security` (HSTS, max-age=2y + includeSubDomains + preload)
- הוספת `Cross-Origin-Opener-Policy: same-origin`
- הוספת `Cross-Origin-Resource-Policy: same-origin`
- שינוי `Permissions-Policy`: microphone הוסר (היה self), נוספו payment=(self), usb=(), magnetometer=(), gyroscope=(), accelerometer=()

> **לא נוסף CSP** — דורש בדיקה זהירה כי עלול לשבור Tailwind/Next inline styles. נשאיר לסבב הבא.

### ✅ תיקון 2 — חיזוק `/api/admin/reset-password`
**קובץ:** [src/app/api/admin/reset-password/route.ts](src/app/api/admin/reset-password/route.ts)
- השוואת secret עם `crypto.timingSafeEqual` (היה `===` — חשוף ל-timing attack)
- אימות `ADMIN_SECRET` ≥ 32 תווים בכל קריאה
- Rate limit חדש ייעודי: **3 בשעה לכל IP** (היה 10 ב-15 דקות)
- אורך סיסמה מינימום הוקשח ל-8 (היה 6)
- מקסימום 200 תווים (DoS guard)
- ולידציית פורמט אימייל
- **Response masking** — אותה תגובה גם אם המשתמש לא נמצא (מונע enumeration)
- Audit log חזק עם IP+userId+email+timestamp לכל איפוס מוצלח

### ✅ תיקון 3 — Kill-switch ל-setup endpoints
**קבצים:**
- [src/app/api/setup/create-admin/route.ts](src/app/api/setup/create-admin/route.ts)
- [src/app/api/setup/make-admin/route.ts](src/app/api/setup/make-admin/route.ts)

שני ה-endpoints מוחזרים כעת **410 Gone** אלא אם `SETUP_ENABLED=true` בסביבה.

ב-`make-admin` נוסף גם:
- timingSafeEqual לבדיקת `SETUP_SECRET`
- אימות אורך מינימום 32 תווים

> **למה kill-switch ולא מחיקה?** למקרה שתצטרך אותם בעתיד לשחזור. סגירה מאחורי flag = ביטול בטוח עם שחזור מיידי.

### ✅ תיקון 4 — Zod ל-`/api/email/send`
**קובץ:** [src/app/api/email/send/route.ts](src/app/api/email/send/route.ts)
- ולידציה מלאה עם Zod
- נושא מקסימום 200 תווים
- חסימת CRLF בנושא (header injection guard)
- תוכן מקסימום 50,000 תווים (DoS guard)

### ✅ תיקון 5 — Open redirect (false positive)
**קובץ:** [src/app/api/auth/google-calendar/callback/route.ts](src/app/api/auth/google-calendar/callback/route.ts)

הסוכן סימן זאת כפגיעות, אך בבדיקה מעמיקה — ה-`redirectUrl` נבנה מ-`NEXTAUTH_URL` (משתנה סביבה, לא מקלט משתמש). **אין open redirect כאן.** מדלג.

### ✅ תיקון 6 — Refund tolerance
**קובץ:** [src/app/api/admin/cardcom/refund/route.ts](src/app/api/admin/cardcom/refund/route.ts)

הסוכן סימן את 0.01 ש"ח tolerance כסכנה. בבדיקה — זה מכוון להתאים לעיגול של Cardcom, ו-Cardcom עצמו דוחה refund גדול מהמקורי. **הסיכון תיאורטי בלבד.** לא משנה.

### ✅ תיקון 7 — Deep sanitization של Cardcom payloads
**קבצים חדשים/משתנים:**
- חדש: [src/lib/cardcom/sanitize.ts](src/lib/cardcom/sanitize.ts) — מודול משותף
- [src/app/api/webhooks/cardcom/admin/route.ts](src/app/api/webhooks/cardcom/admin/route.ts) — שימוש במודול
- [src/app/api/webhooks/cardcom/user/route.ts](src/app/api/webhooks/cardcom/user/route.ts) — sanitize ל-rawPayload (לא היה!)
- [src/lib/cardcom/webhook-claim.ts](src/lib/cardcom/webhook-claim.ts) — sanitize עמוק (היה shallow)

**מה משתנה:** Cardcom payloads שנשמרים ב-DB עוברים deep redact recursive. נחסמו: ApiPassword, ApiName, ApiKey, Password, CVV/CVV2, CardNumber, PAN, FullPan, IdNumber. נשאר: Last4, Token (לחיוב חוזר), Expiry.

### ✅ תיקון 9 — `poweredByHeader: false`
**קובץ:** [next.config.ts](next.config.ts)

### ✅ תיקון 10 — Helper מרכזי `getClientIp`
**קובץ חדש:** [src/lib/get-client-ip.ts](src/lib/get-client-ip.ts)

## 🚨 ממצא קריטי שלי דרך Render API

### CRON_SECRET חשוף מילולית בכל cron job startCommand!
ב-5 ה-cron jobs שלך ב-Render, ה-`startCommand` כולל **את ערך ה-CRON_SECRET באופן מילולי** במקום להשתמש בהפניה ל-`$CRON_SECRET`. דוגמה:
```js
'Authorization':'Bearer ac55cc6bbe29ace5a062d7c7793bcd5c'
```
**זה כנראה איך הפנטסטר נכנס תוך 5 דקות:** מספיק שהיה לו רגע צפייה בדשבורד Render, לוג deploy, או cache של דפדפן. הסוד נראה.

**פעולה דחופה ביותר:**
1. **rotation מיידית של CRON_SECRET** — `openssl rand -hex 32`
2. החלף את ה-`startCommand` של כל cron job ב-Render כך שישתמש ב-`process.env.CRON_SECRET` במקום ערך מילולי. דוגמה:
```js
const headers = {'Authorization':'Bearer '+process.env.CRON_SECRET}
```
3. הגדר `CRON_SECRET` ב-environment variables של ה-cron service (במקום בפקודה).
4. אחר כך bury old value — כל לוגי ה-deploy בעבר עדיין מכילים אותו.

## ❌ דברים שאני **לא** יכול לעשות — צריך אותך

### 🔴 דחוף ביותר — Secret Rotation
**אם איש האבטחה ראה את `.env.local` שלך** או יש לו גישה למחשב — צריך **רוטציה לכל הסודות**:

| Secret | איפה | איך לעשות rotation |
|--------|------|-------------------|
| `DATABASE_URL` | Render dashboard | יצירת DB password חדש ב-Render |
| `NEXTAUTH_SECRET` | Render env | `openssl rand -base64 32` — אזהרה: כל המשתמשים יצטרכו להתחבר מחדש |
| `RESEND_API_KEY` | Resend dashboard | יצירת API key חדש, ביטול הישן |
| `ENCRYPTION_KEY` | Render env | זהירות! משבר נתונים מוצפנים. צריך migration |
| `CRON_SECRET` | Render env | `openssl rand -hex 32` |
| `ADMIN_SECRET` | Render env | `openssl rand -hex 32`, מינימום 32 תווים |
| `SETUP_SECRET` | Render env | `openssl rand -hex 32`, מינימום 32 תווים |
| `GOOGLE_API_KEY` | Google Cloud Console | rotation דרך GCP |
| `CARDCOM_*` | Cardcom dashboard | דרך תמיכת Cardcom |

## ✅ דברים שכבר היו טוב

הסוכנים אישרו שהדברים האלה תקינים:
- IDOR checks על clients/sessions/payments (`where: { id, therapistId: userId }`)
- bcrypt עם constant-time compare
- NextAuth + JWT + SameSite (CSRF protection מובנה)
- `requireAuth/requirePermission` בכל admin routes
- booking endpoint עם honeypot ו-rate limit
- קבלות פומביות עם token (24 chars HMAC)
- `verifyReceiptToken` עם timingSafe
- cron-auth עם timingSafeEqual
- Cardcom webhook signature + timestamp anti-replay (±5 דק)
- Refund atomic claim עם Decimal precision
- `claimWebhook` lease-based idempotency
- Withaudit transactional pattern

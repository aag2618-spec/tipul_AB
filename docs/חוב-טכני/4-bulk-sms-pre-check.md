# 4. pre-check ב-sendBulkSMS — מניעת כשלים חלקיים

**מקור:** ביקורת Cursor — Stage 1.17, M-Bulk
**עדיפות:** בינונית (רק רלוונטי כשמשתמשים בשליחה מרובה)
**סטטוס:** פתוח

---

## מה הבעיה

הפונקציה `sendBulkSMS` מקבלת רשימה של הודעות ושולחת אותן אחת אחרי השנייה.

### תרחיש הבעיה (כש-`USE_NEW_CONSUME_SMS=true`)

נניח:
- משתמש שולח **100 הודעות** במכה אחת.
- ה-quota שלו הוא **10 הודעות בלבד**.

מה יקרה:
- 10 ההודעות הראשונות יישלחו (כי `consumeSms` יצליח).
- 90 ההודעות הבאות ייפלו עם שגיאת `QuotaExhaustedError` — **אחת אחרי השנייה**.
- כלומר: **90 קריאות לחינם** לבסיס הנתונים, רק כדי לקבל את אותה שגיאה 90 פעם.

זה לא שובר את המערכת — אבל זה בזבזני, איטי, ועלול להציף לוגים.

---

## למה זה לא דחוף

- שני ה-feature flags **כבויים כרגע** — אז התרחיש בכלל לא מתרחש.
- גם כשיהיו דלוקים — רק משתמשים שעושים שליחה מרובה גדולה ייתקלו בבעיה.
- אין נזק אמיתי — רק חוסר יעילות.

---

## פתרון מוצע

הוספת פונקציה `consumeSmsBatch(userId, count)` שעושה **בדיקה ב-batch אטומית**:

```typescript
async function consumeSmsBatch(userId: string, count: number) {
  // 1. בודק במכה אם יש מספיק קרדיט ל-N הודעות
  // 2. אם כן: deduct פעם אחת על כל ה-N
  // 3. אם לא: זורק QuotaExhaustedError לפני שום שליחה
}
```

ואז ב-`sendBulkSMS`:

```typescript
// לפני הלולאה:
await consumeSmsBatch(userId, messages.length);
// אם הגענו לכאן — יש מספיק קרדיט. עכשיו שולחים.
for (const msg of messages) { /* send */ }
```

### אופציה מתקדמת (אם בא לך)

שליחה חלקית של ההודעות שכן נכנסות בתקציב + החזרת error מסודר ללקוח שאומר: "נשלחו 10 מתוך 100, כי לא היה מספיק קרדיט".

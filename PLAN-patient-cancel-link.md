# תוכנית: ביטול תור ע"י מטופל דרך קישור אישי

## מטרה
לתת למטופל אפשרות לבקש ביטול תור דרך קישור אישי שנשלח אליו (מייל/SMS), בלי
התחברות/חשבון. הלחיצה (אחרי קוד אימות) פותחת עמוד "הפגישות שלי" עם כל התורים
הקרובים, והמטופל בוחר פגישה לבטל → נוצרת `CancellationRequest` קיימת → כל מנגנון
הניהול הקיים (דף בקשות, אישור/דחייה, מיילים, פעמון) ממשיך לעבוד מאליו.

## רקע שקבע את התכנון
- **למטופל אין חשבון/התחברות.** הוא קיים רק כרשומת `Client`. הדלת היחידה שלו
  למערכת היא קישור אישי בהודעה — כמו זימון עצמי / תשלום / שאלון היום.
- **פגישות חוזרות נוצרות כ-N רשומות נפרדות.** הדיאלוג "פגישה חדשה" מאפשר שכפול
  ל-2/4/8/12/16 שבועות → לולאה שיוצרת פגישה אמיתית לכל שבוע. לכן קישור-לכל-פגישה
  לא מעשי (המטופל יצטרך לחפש את ההודעה של אותה פגישה).

## החלטות שהתקבלו (המשתמש, 2026-06-30)
1. **בקשה לאישור** — הלחיצה שולחת בקשה למטפל; הוא מאשר/דוחה (לא ביטול אוטומטי).
2. **קישור אישי לכל מטופל (לא לפגישה)** → עמוד "הפגישות שלי" עם רשימת כל התורים
   הקרובים. פותר גם את הסדרה (בוחר מהרשימה) וגם את "לחפש הודעה ישנה" (קישור יציב).
3. **קוד אימות (OTP) — מייל או SMS.** מאחר שהעמוד חושף את כל לוח הפגישות, נדרש קוד.
   מנגנון קיים מהזימון העצמי: שולח במייל אם יש, אחרת SMS; כבר חוסם בשבת.
4. **ביטול פגישה בודדת בכל פעם** — אין "בטל את כל הסדרה". הרשימה מציגה הכל,
   הביטול חל על אחת בכל לחיצה.
5. **שבת**: SMS לא נשלח בשבת (המתזמן + sendSMS כבר מדלגים); עמוד הביטול וה-OTP
   נחסמים בשבת/חג ומציגים "שבת שלום, נא לנסות במוצ"ש".
6. **הקישור בכל הודעה, יציב** — נכנס למייל האישור (בעת קביעת התור) וגם לכל תזכורת.
   כך המטופל מקבל אותו מוקדם ולא תלוי בתזכורת שעלולה להגיע אחרי שחלון הביטול נסגר.

## עיקרון: איך המטופל מגיע לעמוד
אין "מקום" שהמטופל נכנס אליו — רק קישור בהודעה:
| מתי | מה נשלח | מוביל ל |
|-----|---------|---------|
| בעת קביעת התור | מייל אישור + קישור "הפגישות שלי" | אותו עמוד יציב |
| 24ש' / שעתיים לפני | תזכורת (מייל/SMS) + אותו קישור | אותו עמוד יציב |
המטופל פותח את ההודעה האחרונה, לוחץ → קוד אימות → רואה את כל הפגישות → בוחר לבטל.
כל פגישה אוכפת בנפרד את חלון הביטול שלה (אפשר לבטל שבוע 12 מראש; אי אפשר את מחר
אם עברנו את החלון).

## רכיבים לבנייה

### 1. קישור אישי + OTP (מיחזור תשתית `BookingLink`)
`BookingLink` כבר קיים: per-client, טוקן 256 ביט, תוקף 60 יום, OTP מייל/SMS,
חסימת שבת, rate-limit, נעילה אחרי כשלים. נשכפל/נרחיב את הדפוס לעמוד הפגישות.
החלטת מימוש (בעת הבנייה): להשתמש באותו טוקן `BookingLink`, או טוקן ייעודי נפרד
לניהול-פגישות (כדי לא לכפות שזימון-עצמי יהיה דלוק). ברירת מחדל מוצעת: טוקן ייעודי
שמשכפל את אותה לוגיקה (`src/lib/booking-links.ts` — `evaluateOtpSend/Attempt`).

### 2. עמוד ציבורי "הפגישות שלי"
`src/app/p/appointments/[token]/page.tsx` (+ רכיב client).
מבנה במתכונת `src/app/p/` הקיים. זרימה: טעינת טוקן → אם לא מאומת, מסך "שלח קוד"
→ הזנת קוד → רשימת פגישות קרובות (תאריך, שעה, מטפל) + כפתור "בקש ביטול" לכל אחת.
מצבים לכל פגישה: ניתן לביטול / חלון חסום ("צרו קשר") / כבר יש בקשה ממתינה.
**שבת/חג**: כל העמוד חסום ("שבת שלום, נסה במוצ"ש"). מינימום PHI: מטפל+מועד בלבד.

### 3. נתיב API ציבורי
`src/app/api/p/appointments/[token]/route.ts` — **בלי `requireAuth`**:
- `send-otp` / `verify-otp` — מיחזור הלוגיקה מ-`booking/t/[token]`.
- `list` — אחרי אימות: מחזיר פגישות קרובות (SCHEDULED) של המטופל, שדות אדמין בלבד.
- `cancel` (POST, פגישה בודדת): אוכף `minCancellationHours` + `allowClientCancellation`
  + שבת → יוצר `CancellationRequest` + מעדכן פגישה ל-`PENDING_CANCELLATION` +
  `cancellationRequestedAt` → התראה+מייל למטפל (מיחזור `request-cancellation/route.ts`).
- rate-limit + 404 אחיד (anti-enumeration).

### 4. כפתור/קישור בתבניות (אישור + תזכורת)
`src/lib/email-templates.ts` — `createSessionConfirmationEmail` (העיקר),
`create24HourReminderEmail`, `create2HourReminderEmail`, `createManualSessionReminderEmail`
מקבלים `manageUrl?: string` ומציגים קישור רק אם הועבר. תבנית SMS: שורה
`לניהול/ביטול: {קישור}`. נקודות השליחה מייצרות/שולפות טוקן ובונות
`manageUrl = ${NEXTAUTH_URL}/p/appointments/${token}`:
- אישור: `sessions/route.ts`, `sessions/send-confirmation`, `booking/t/[token]`, `cron/booking-outbox`.
- תזכורות: `cron/reminders` + `cron/reminders-2h`.
**מותנה ב-`allowClientCancellation`** — כבוי → לא מצרפים קישור.

## תיקון נלווה (חובה)
המלבן "מה דורש טיפול" + ה-badge ב-`SessionRow` מחפשים `status: "SCHEDULED"`,
אבל בקשה אמיתית נותנת `PENDING_CANCELLATION` → המלבן תמיד ריק. לתקן ב-
`src/components/dashboard/secretary-home.tsx` (שורות 153, 255, 264) ל-`PENDING_CANCELLATION`.

## מה כבר קיים — מיחזור מלא
- ✅ `CancellationRequest` + דף `/dashboard/cancellation-requests` + approve/reject + מיילים.
- ✅ `minCancellationHours` + `allowClientCancellation` (`CommunicationSetting`).
- ✅ `BookingLink` + OTP מייל/SMS + חסימת שבת + rate-limit (`booking-links.ts`).
- ✅ מבנה עמודי `/p/` + מתזמן תזכורות + `CommunicationLog`/dedup + פעמון (`Notification`).

## אבטחה / PHI
- טוקן 256 ביט, per-client, תוקף מוגבל. OTP (מייל/SMS) לפני חשיפת הרשימה.
- העמוד חושף מינימום PHI (מטפל+מועד). אכיפת חלון+מתג+שבת בשרת (לא רק הסתרת UI).
- rate-limit + 404 אחיד. ביקורת אבטחה (security-review) לפני פוש — נתיב ציבורי חדש.

## שלבי ביצוע
1. תיקון המלבן (`PENDING_CANCELLATION`) — קטן, עצמאי, בטוח.
2. Schema: טוקן ניהול-פגישות (שדות/מודל) + `prisma db push`.
3. נתיב API ציבורי: send-otp / verify-otp / list / cancel (מיחזור booking + request-cancellation).
4. עמוד `/p/appointments/[token]` + רכיב טופס (OTP → רשימה → ביטול).
5. תבניות: `manageUrl` אופציונלי באישור + תזכורות.
6. נקודות שליחה: יצירת/שליפת טוקן + העברת `manageUrl` (מותנה במתג).
7. ביקורת אבטחה + בדיקות.

## בדיקות
- tsc + build.
- יחידה: אכיפת חלון, מתג כבוי, שבת, טוקן פג, OTP (שליחה/אימות/נעילה),
  ביטול פגישה שכבר בוטלה / שכבר יש לה בקשה.
- ידני: אישור/תזכורת → קישור → קוד → רשימה → ביטול אחת → הופעה בדף הבקשות + במלבן
  + פעמון → אישור המטפל → מייל למטופל.

## פתוחים (הוכרעו רובם)
- שבת — הוכרע (חסימה מלאה + אין SMS).
- סדרה — הוכרע (רשימה, ביטול בודד בכל פעם).
- אימות — הוכרע (OTP מייל/SMS).
- נותר למימוש: טוקן ייעודי מול מיחזור `BookingLink` (יוכרע בבנייה).

## 📊 תקציר מנהלים

המערכת **מוגנת היטב ברוב המעטפות** (rate-limit, timing-safe, 2FA לתפקידים בכירים, CSP אכוף, encryption AES-256-GCM, scope helpers ארכיטקטוניים). אך זוהו **בערך 70 ממצאים**, מתוכם **9 ברמת קריטי** המחייבים תיקון מיידי לפני production.

### חלוקה לפי חומרה

| חומרה | כמות | משמעות |
|---|---|---|
| 🔴 **קריטי** | 9 | סיכון מיידי לדליפת PHI / privilege escalation / overwrite cross-tenant |
| 🟠 **גבוה** | 19 | bypass הרשאות, חוסר session invalidation, PII לצד שלישי |
| 🟡 **בינוני** | 20 | info disclosure, חוסר validation, race conditions |
| 🔵 **נמוך** | 22 | hardening, עקביות, UX-defensive |

### הממצא המסוכן ביותר במכלול

**ה-AI endpoints (`/api/analyze/note`, `/api/analyze/summary`, `/api/ai/session/analyze`, `/api/ai/questionnaire/analyze-single`) מהווים נקודת תורפה כפולה:**
1. **IDOR:** מסננים רק לפי `therapistId === userId` במקום ב-`buildClientWhere`/`buildSessionWhere` — מטפל מכל ארגון יכול לשלוח `clientId` של מטופל מארגון אחר ולקרוא/לכתוב את שדה ה-`comprehensiveAnalysis` שלו.
2. **PII ל-Google:** שם המטופל המלא, תאריך לידה, היסטוריה רפואית, הקשר תרבותי ותשובות גולמיות לשאלוני אבחון — נשלחים ישירות ל-Google Gemini **בלי pseudonymization** וללא DPA ייעודי שראיתי.

### C1. Privilege Escalation: MANAGER יוצר ADMIN דרך POST /api/admin/users

**קובץ:** [src/app/api/admin/users/route.ts:81-145](src/app/api/admin/users/route.ts#L81-L145)

```ts
const { name, email, password, phone, role } = body;
...
data: { ..., role: role || 'USER' }
```

ה-endpoint מוגן רק ב-permission `users.create` (גם MANAGER מקבל). אין allowlist על `role`. **MANAGER יכול לשלוח `role: "ADMIN"` ולקבל ADMIN שלם**, עוקף את `users.change_role` שאמור להגן על זה.

**ניצול:**
```
POST /api/admin/users
{ "email": "evil@x.com", "password": "...", "role": "ADMIN" }
```

**תיקון:** Zod schema עם `role: z.enum(["USER","MANAGER"])`; ליצירת ADMIN/CLINIC_OWNER לדרוש permission נוסף `users.change_role`.

### C8. Google OAuth Account Linking — Takeover אוטומטי על משתמשים לא verified

**קובץ:** [src/lib/auth.ts:216-302](src/lib/auth.ts#L216-L302)

ה-`signIn` callback של Google מקשר אוטומטית חשבון Google לחשבון email קיים, גם אם `existingUser.emailVerified === null`.

**תרחיש exploit:**
1. תוקף יודע על משתמש legacy `victim@example.com` שלא אומת ב-MyTipul.
2. תוקף יוצר חשבון Gmail עם email זהה (אם הדומיין מאפשר, או דרך alias).
3. תוקף עושה login דרך Google → email match → קישור אוטומטי → תוקף נכנס לחשבון של ה-victim ב-MyTipul.

ההגנה היחידה היא mail אזהרה ל-victim אחרי הקישור — reactive, לא preventative.

**תיקון:**
- אם `existingUser.emailVerified === null` → לדחות linking, לדרוש verification דרך MyTipul.
- אופציה: confirmation flow אינטראקטיבי (קליק במייל לאישור linking).

### H15. אין rate-limit per-email על register + forgot-password

### H16. Impersonator יכול להמשיך לפעול אחרי שאיבד role=OWNER

**קובץ:** [src/lib/impersonation.ts:32-81](src/lib/impersonation.ts#L32)

**תיקון:** הוספת בדיקה `if (impersonator.role !== "CLINIC_OWNER" && impersonator.clinicRole !== "OWNER") return null;`

### H17. recordings file storage — אין signed URLs

**קובץ:** [src/app/api/uploads/[...path]/route.ts:113-128](src/app/api/uploads/%5B...path%5D/route.ts#L113)

UUID v4 לא ניחוש מעשי, אבל ההגנה היחידה היא DB lookup. אם הקובץ נחשף בעקיפין (referrer leak, screenshot, cache), כל מי שמחזיק את ה-URL מקבל גישה. ברגישות הזו של recordings (תכנית-שיחה!) — נדרשות signed URLs עם TTL קצר.

**תיקון:** מעבר ל-signed URLs HMAC עם expiry של 10 דקות; refresh ב-client לפני expiry.

**תיקון:** generate 10 backup codes ב-2FA setup; hash + single-use.

## ✅ נקודות חזק במערכת (תיעוד)

1. **bcrypt.compare עם dummy hash** — אין timing-based enumeration ב-login.
2. **timingSafeEqual בכל ה-cron + Render + Pulseem** — `bearerEquals` עקבי.
3. **Cardcom GetLpResult callback re-verify** — לא סומך על body חתום.
4. **PCI compliance** — אין שמירת PAN/CVV; רק last4 + token + expiry.
5. **CSP enforced** + HSTS preload + X-Frame-Options DENY + frame-ancestors none.
6. **2FA gate ב-API routes** מחוץ ל-middleware (defense-in-depth).
7. **Impersonation** מאומת ב-server (client לא יכול לזייף `actingAs.role`).
8. **`withAudit` transactional** עם retry על serialization/deadlock.
9. **Cardcom payload deep sanitization** — redact של ApiPassword/CVV/PAN לפני שמירה.
10. **Rate-limiting מרובד** — IP + global + user + tier ב-middleware.
11. **AES-256-GCM** עם IV+salt אקראי 16 byte; authTag מאומת.
12. **`/api/recordings/[id]`, `/documents/[id]`, `/consent-forms/*`, `/questionnaires/responses/*`** — כולם משתמשים נכון ב-scope helpers עם logDataAccess.

14. **NEXTAUTH_SECRET strength validated** ב-startup; ENCRYPTION_KEY validated ב-prod.
